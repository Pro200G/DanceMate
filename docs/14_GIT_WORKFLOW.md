# 14_GIT_WORKFLOW.md

# Git Workflow

**Проект:** DanceMate

**Версия:** 1.0

---

# 1. Назначение

Документ определяет единый процесс работы с системой контроля версий Git.

Цели:

* единый процесс разработки;
* безопасное внесение изменений;
* удобное сопровождение проекта;
* автоматизация релизов;
* предотвращение конфликтов;
* обеспечение стабильности Production.

Все участники проекта обязаны соблюдать данный Workflow.

---

# 2. Используемая модель

Проект использует **Git Flow** с элементами **Trunk-Based Development**.

Основные принципы:

* стабильная ветка Production;
* отдельная ветка разработки;
* короткоживущие feature-ветки;
* обязательный Pull Request;
* обязательный Code Review;
* автоматическая проверка через CI.

---

# 3. Структура веток

```text
main
develop
feature/*
bugfix/*
hotfix/*
release/*
experiment/*
```

---

# 4. Назначение веток

## main

Содержит только стабильные версии.

Особенности:

* только рабочий код;
* соответствует Production;
* запрещены прямые коммиты.

---

## develop

Основная ветка разработки.

Особенности:

* объединяет завершённые функции;
* используется для подготовки релизов.

---

## feature/*

Разработка новой функциональности.

Примеры:

```text
feature/profile-search

feature/chat-websocket

feature/payment-system

feature/event-calendar
```

---

## bugfix/*

Исправление ошибок.

Примеры:

```text
bugfix/chat-scroll

bugfix/profile-validation

bugfix/event-filter
```

---

## hotfix/*

Срочные исправления Production.

Примеры:

```text
hotfix/payment-crash

hotfix/jwt-validation
```

---

## release/*

Подготовка новой версии.

Пример:

```text
release/v1.3.0
```

Разрешены только:

* исправление ошибок;
* обновление документации;
* изменение версии.

---

## experiment/*

Экспериментальные исследования.

Никогда не попадают напрямую в Production.

---

# 5. Жизненный цикл задачи

```text
Issue
    │
    ▼
feature/*
    │
    ▼
Pull Request
    │
    ▼
Code Review
    │
    ▼
CI/CD
    │
    ▼
develop
    │
    ▼
release/*
    │
    ▼
main
```

---

# 6. Правила создания веток

Каждая задача создаётся из:

```text
develop
```

Например:

```bash
git checkout develop

git pull

git checkout -b feature/profile-search
```

---

# 7. Pull Request

Каждое изменение оформляется через Pull Request.

Запрещается:

* push напрямую в main;
* push напрямую в develop.

---

# 8. Требования к Pull Request

Pull Request должен содержать:

* описание изменений;
* ссылку на задачу;
* список изменённых компонентов;
* сведения о миграциях;
* сведения о новых API;
* сведения об изменениях БД.

---

# 9. Code Review

Минимум один разработчик обязан проверить:

* архитектуру;
* SOLID;
* DDD;
* тесты;
* безопасность;
* производительность;
* читаемость.

Для изменений безопасности или базы данных рекомендуется два независимых ревью.

---

# 10. CI Pipeline

Перед объединением автоматически выполняются:

1. Ruff
2. Black
3. isort
4. mypy
5. Bandit
6. Unit Tests
7. Integration Tests
8. API Tests
9. Проверка миграций Alembic
10. Сборка Docker

При любой ошибке объединение запрещается.

---

# 11. Conventional Commits

Все коммиты оформляются по стандарту Conventional Commits.

Примеры:

```text
feat(profile): add recommendation engine

fix(chat): resolve websocket reconnect

refactor(events): simplify event filtering

docs(database): update ER diagram

test(payments): add integration tests

perf(search): optimize profile lookup

ci(github): update workflow

build(docker): upgrade base image
```

---

# 12. Версионирование

Используется Semantic Versioning (SemVer).

Формат:

```text
MAJOR.MINOR.PATCH
```

Примеры:

```text
1.0.0
1.1.0
1.1.3
2.0.0
```

---

# 13. Работа с миграциями

Каждое изменение PostgreSQL сопровождается миграцией Alembic.

Порядок:

1. изменение моделей;
2. создание миграции;
3. проверка миграции;
4. обновление SQL-проекта (при необходимости);
5. обновление документации.

Запрещается изменять существующие миграции после их применения.

---

# 14. Работа с документацией

При изменении:

* архитектуры;
* базы данных;
* API;
* безопасности;
* структуры проекта

обязательно обновляются соответствующие документы в каталоге `docs/`.

---

# 15. Работа с тестами

Любая новая функциональность сопровождается:

* Unit Tests;
* Integration Tests (если применимо);
* API Tests (для новых эндпоинтов).

Исправление ошибки должно включать тест, воспроизводящий найденный дефект.

---

# 16. Работа с релизами

Порядок выпуска:

1. создание `release/*`;
2. исправление найденных ошибок;
3. прохождение полного CI;
4. обновление номера версии;
5. объединение в `main`;
6. создание Git Tag;
7. публикация Release;
8. объединение изменений обратно в `develop`.

---

# 17. Работа с Hotfix

Порядок:

```text
main
    │
    ▼
hotfix/*
    │
    ▼
Pull Request
    │
    ▼
main
    │
    ├────► Tag
    │
    └────► develop
```

Hotfix обязательно переносится обратно в `develop`.

---

# 18. Git Tags

Каждый релиз помечается тегом.

Пример:

```text
v1.0.0

v1.1.0

v1.2.5
```

Теги используются для:

* развёртывания;
* отката;
* формирования Release Notes.

---

# 19. Release Notes

Каждый релиз сопровождается описанием:

* новых функций;
* исправленных ошибок;
* изменений API;
* изменений БД;
* изменений миграций;
* изменений безопасности.

---

# 20. Защита веток

Для веток `main` и `develop` включается защита:

* запрет force push;
* обязательный Pull Request;
* обязательный успешный CI;
* обязательное Code Review;
* обязательное разрешение конфликтов перед слиянием.

---

# 21. Правила работы с GitHub

Используются:

* Issues;
* Pull Requests;
* Discussions;
* Projects;
* Milestones;
* Releases;
* GitHub Actions.

Все изменения связываются с соответствующей задачей (Issue).

---

# 22. Автоматизация

GitHub Actions автоматически выполняет:

* проверку качества кода;
* запуск тестов;
* сборку Docker-образов;
* проверку миграций;
* публикацию документации (при необходимости);
* подготовку артефактов релиза.

---

# 23. Политика отката

Каждый релиз должен иметь возможность безопасного отката.

Откат включает:

* возврат к предыдущему Git Tag;
* откат контейнеров;
* восстановление базы данных (если требуется);
* проверку работоспособности.

---

# 24. Контроль качества

Перед объединением изменений должны быть выполнены:

* все тесты успешны;
* покрытие соответствует требованиям;
* документация обновлена;
* миграции проверены;
* код соответствует стандартам;
* отсутствуют критические замечания Code Review.
