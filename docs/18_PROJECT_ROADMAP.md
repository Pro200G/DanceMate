# 18_PROJECT_ROADMAP.md

# Project Roadmap

**Проект:** DanceMate
**Версия:** 1.0

---

# 1. Назначение

Документ определяет стратегическое развитие проекта DanceMate.

Он связывает:

* бизнес-цели;
* техническую архитектуру;
* этапы разработки;
* релизный цикл;
* масштабирование системы.

---

# 2. Видение продукта

DanceMate — это социальная платформа для танцевального сообщества, включающая:

* поиск партнёров;
* мероприятия;
* чаты;
* рейтинги;
* групповые активности;
* медиа;
* систему приглашений;
* монетизацию.

---

# 3. Этапы развития проекта

## Этап 1 — MVP (Foundation)

Цель: базовая работа платформы.

Функции:

* регистрация и авторизация;
* профили пользователей;
* базовые танцевальные стили;
* приглашения;
* простые мероприятия;
* базовый чат;
* рейтинги.

Технически:

* монолит FastAPI;
* PostgreSQL;
* Redis (минимально);
* MinIO (базово);
* Docker Compose.

---

## Этап 2 — Social Core

Цель: формирование социальной активности.

Функции:

* расширенные чаты;
* WebSocket messaging;
* групповые сборы (Squads);
* расширенные профили;
* рекомендации пользователей;
* уведомления;
* медиа-контент.

---

## Этап 3 — Growth & Engagement

Цель: удержание пользователей и рост активности.

Функции:

* система рейтингов;
* система благодарностей;
* алгоритмы рекомендаций;
* умные уведомления;
* активность событий;
* аналитика поведения.

---

## Этап 4 — Monetization

Цель: монетизация платформы.

Функции:

* подписки;
* платные события;
* премиум-профили;
* рекламные механизмы;
* аналитика доходов.

---

## Этап 5 — Scale

Цель: масштабирование системы.

Функции:

* горизонтальное масштабирование backend;
* распределённые workers;
* CDN для медиа;
* оптимизация PostgreSQL;
* разделение сервисов (microservices-ready);
* Kubernetes-ready инфраструктура.

---

# 4. Архитектурная эволюция

## Фаза 1 — Modular Monolith

* один backend;
* четкое разделение слоёв (DDD);
* единая база данных.

---

## Фаза 2 — Service-Oriented Modules

* выделение отдельных доменов;
* независимые workers;
* расширение очередей задач.

---

## Фаза 3 — Microservices Ready

* возможность отделения сервисов:

  * chat-service;
  * event-service;
  * user-service;
  * payment-service;
* единый API Gateway.

---

# 5. Технический долг

Отслеживается:

* медленные SQL-запросы;
* отсутствие индексов;
* дублирование бизнес-логики;
* отсутствие тестов;
* устаревшие зависимости;
* узкие места в WebSocket.

Каждый релиз должен содержать:

* список debt items;
* приоритет исправления.

# 10. Масштабирование системы

## Backend

* горизонтальное масштабирование;
* stateless API;
* балансировка через Nginx.

---

## Database

* read replicas;
* индексация;
* партиционирование таблиц (при росте данных).

---

## Cache

* Redis cluster-ready;
* распределённое кеширование.

---

## Media

* MinIO + CDN;
* подписанные URL.

---

# 11. Этапы релиза архитектуры

## Stage 1

* монолит;
* единая база;
* Docker Compose.

---

## Stage 2

* разделение worker-ов;
* усиление кеширования;
* observability полный стек.

---

## Stage 3

* сервисная декомпозиция;
* API Gateway;
* event-driven архитектура.

---

# 12. Риски проекта

| Риск               | Решение                        |
| ------------------ | ------------------------------ |
| Рост нагрузки      | горизонтальное масштабирование |
| медленные SQL      | индексы + оптимизация          |
| рост сложности     | модульная архитектура          |
| WebSocket нагрузка | отдельные workers              |
| хранение медиа     | MinIO + CDN                    |

---

# 13. Долгосрочная стратегия

* переход к event-driven architecture;
* внедрение message broker (Kafka/RabbitMQ);
* выделение микросервисов;
* геораспределение инфраструктуры;
* multi-region deployment.

---

# 14. Definition of Done (DoD)

Функция считается завершённой, если:

* написан код;
* написаны тесты;
* обновлена документация;
* проведён Code Review;
* пройдён CI/CD;
* проверена производительность;
* проверена безопасность.

---

# 15. Definition of Ready (DoR)

Задача готова к разработке, если:

* есть описание;
* определены acceptance criteria;
* понятна архитектурная зона;
* оценены зависимости.
