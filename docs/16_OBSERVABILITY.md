# Observability

**Проект:** DanceMate

**Версия:** 1.0

---

# 1. Назначение

Документ определяет архитектуру наблюдаемости (Observability) приложения DanceMate.

Цели:

* контроль состояния системы;
* раннее обнаружение ошибок;
* анализ производительности;
* аудит действий пользователей;
* диагностика инцидентов;
* контроль доступности сервисов.

Observability является обязательной частью Production-инфраструктуры.

---

# 2. Компоненты Observability

Архитектура включает:

* централизованное логирование;
* сбор метрик;
* распределённую трассировку;
* аудит;
* мониторинг инфраструктуры;
* Health Checks;
* систему оповещений.

---

# 4. Централизованное логирование

Все сервисы используют единый механизм логирования.

Логи формируются в формате JSON.

Каждая запись содержит:

* Timestamp;
* Level;
* Service;
* Environment;
* Request ID;
* Correlation ID;
* User ID (если авторизован);
* HTTP Method;
* URL;
* Duration;
* Message.

---

# 5. Уровни логирования

Используются стандартные уровни:

| Уровень  | Назначение             |
| -------- | ---------------------- |
| DEBUG    | Диагностика разработки |
| INFO     | Нормальная работа      |
| WARNING  | Потенциальные проблемы |
| ERROR    | Ошибки обработки       |
| CRITICAL | Критические сбои       |

В Production уровень `DEBUG` отключён.

---

# 6. Категории логов

Выделяются отдельные категории:

* Application Logs;
* Access Logs;
* SQL Logs;
* Security Logs;
* Audit Logs;
* Background Jobs Logs;
* WebSocket Logs.

---

# 7. Метрики

Собираются следующие группы метрик.

## API

* количество запросов;
* время ответа;
* процент ошибок;
* количество активных соединений.

---

## PostgreSQL

* активные подключения;
* длительные запросы;
* использование индексов;
* блокировки;
* размер базы данных.

---

## Redis

* использование памяти;
* количество ключей;
* задержка;
* количество подключений.

---

## MinIO

* объём хранилища;
* количество объектов;
* скорость загрузки;
* скорость скачивания.

---

## Workers

* очередь задач;
* среднее время выполнения;
* ошибки фоновых задач;
* количество активных процессов.

---

# 8. Метрики приложения

Контролируются:

* регистрация пользователей;
* создание профилей;
* отправка приглашений;
* создание мероприятий;
* сообщения чатов;
* загрузка фотографий;
* оформление подписок.

Эти показатели используются для анализа использования системы.

---

# 9. Prometheus

Prometheus собирает:

* системные метрики;
* метрики FastAPI;
* PostgreSQL;
* Redis;
* MinIO;
* Docker.

Экспорт осуществляется через `/metrics`.

---

# 10. Grafana

Grafana используется для визуализации.

Основные панели:

* API Dashboard;
* Database Dashboard;
* Redis Dashboard;
* Docker Dashboard;
* User Activity Dashboard;
* Business Metrics Dashboard.

---

# 11. Distributed Tracing

Используется OpenTelemetry.

Каждый запрос получает:

* Trace ID;
* Span ID;
* Correlation ID.

Трассировка охватывает:

* API;
* Application Layer;
* PostgreSQL;
* Redis;
* MinIO;
* внешние сервисы.

---

# 12. Correlation ID

Каждый HTTP-запрос получает уникальный идентификатор.

Correlation ID передаётся:

* между сервисами;
* в логи;
* в фоновые задачи;
* в WebSocket.

Это позволяет проследить полный путь обработки запроса.

---

# 13. Health Checks

Все сервисы реализуют проверки состояния.

Проверяются:

* FastAPI;
* PostgreSQL;
* Redis;
* MinIO;
* Background Workers;
* WebSocket.

Основные маршруты:

```text id="4r3zmt"
/health/live
/health/ready
/health/startup
```

---

# 14. Audit Logging

Журнал аудита содержит:

* вход пользователя;
* изменение профиля;
* изменение ролей;
* удаление данных;
* действия администратора;
* платёжные операции.

Записи аудита не подлежат изменению.

---

# 15. Security Monitoring

Контролируются:

* неудачные входы;
* массовые запросы;
* попытки подбора OTP;
* ошибки JWT;
* подозрительная активность;
* превышение Rate Limit.

При выявлении угроз создаётся инцидент.

---

# 16. Alerts

Автоматические оповещения создаются при:

* недоступности API;
* высокой загрузке CPU;
* нехватке памяти;
* переполнении диска;
* недоступности PostgreSQL;
* недоступности Redis;
* ошибках миграций;
* росте количества ошибок HTTP 5xx.

---

# 17. SLA и SLO

Целевые показатели:

| Метрика                  | Значение |
| ------------------------ | -------: |
| Доступность API          |  ≥ 99.9% |
| Среднее время ответа     | ≤ 300 мс |
| 95-й перцентиль          | ≤ 700 мс |
| Ошибки HTTP 5xx          |     < 1% |
| Успешность фоновых задач |    ≥ 99% |

---

# 18. Хранение логов

Политика хранения:

| Тип              | Срок    |
| ---------------- | ------- |
| Application Logs | 30 дней |
| Access Logs      | 30 дней |
| Audit Logs       | 1 год   |
| Security Logs    | 1 год   |
| Error Logs       | 90 дней |

Архивирование выполняется автоматически.

# 22. Правила разработки

Каждый новый компонент обязан:

* публиковать метрики;
* использовать централизованное логирование;
* поддерживать Correlation ID;
* реализовывать Health Checks;
* поддерживать распределённую трассировку.

---

# 23. Эксплуатационные проверки

Перед каждым релизом проверяются:

* доступность Dashboard;
* получение логов;
* корректность метрик;
* работа алертов;
* состояние Health Checks.

---

# 24. Интеграция с CI/CD

После успешного развёртывания автоматически выполняются:

1. Проверка `/health`.
2. Проверка `/metrics`.
3. Проверка доступности Grafana.
4. Проверка подключения к PostgreSQL.
5. Проверка Redis.
6. Проверка MinIO.

При ошибке развёртывание считается неуспешным.
