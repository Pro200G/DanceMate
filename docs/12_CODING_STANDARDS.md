# Coding Standards

**Проект:** DanceMate
**Версия:** 1.0

---

# 1. Назначение

Документ определяет единые стандарты разработки программного кода проекта DanceMate.

Цели:

* единый стиль разработки;
* повышение читаемости кода;
* снижение количества ошибок;
* упрощение сопровождения;
* обеспечение соответствия принципам Clean Architecture, DDD и SOLID.

Все разработчики обязаны соблюдать данные требования.

---

# 2. Основные принципы

Код должен соответствовать следующим принципам:

* SOLID;
* DRY (Don't Repeat Yourself);
* KISS (Keep It Simple, Stupid);
* YAGNI (You Aren't Gonna Need It);
* Clean Code;
* Clean Architecture;
* Domain-Driven Design.

---

# 3. Структура проекта

Запрещается нарушать архитектуру слоёв.

Допустимые зависимости:

```text
API Layer
      │
      ▼
Application Layer
      │
      ▼
Domain Layer
      ▲
      │
Infrastructure Layer
```

Domain Layer не зависит ни от одного другого слоя.

---

# 4. Стиль кода

Используется:

* Python 3.13;
* PEP 8;
* Black;
* Ruff;
* isort.

Максимальная длина строки:

```text
100 символов
```

Используются 4 пробела для отступов.

---

# 5. Типизация

Вся новая логика должна использовать строгую типизацию.

Пример:

```python
def create_profile(command: CreateProfileCommand) -> Profile:
    ...
```

Запрещается использовать `Any`, кроме случаев, когда это технически невозможно.

---

# 6. Документирование

Каждый публичный класс и метод должны иметь docstring.

Стандарт:

```python
def send_invitation(...) -> Invitation:
    """
    Отправляет приглашение пользователю.

    Args:
        ...

    Returns:
        Invitation
    """
```

---

profile_service.py

ProfileService

DEFAULT_PAGE_SIZE

# 8. Импорты

Порядок импортов:

1. стандартная библиотека Python;
2. сторонние библиотеки;
3. внутренние модули проекта.

Пример:

```python
from datetime import datetime

from fastapi import APIRouter

from app.domain.entities import User
```

Используется `isort`.

---

# 9. Организация классов

Порядок элементов класса:

1. константы;
2. конструктор;
3. публичные методы;
4. защищённые методы;
5. приватные методы.

---

# 10. Исключения

Используются только специализированные исключения.

Запрещается:

```python
raise Exception(...)
```

Допустимо:

```python
raise ProfileNotFoundError(...)
```

---

# 11. Логирование

Используется модуль `logging`.

Не допускается:

```python
print(...)
```

Логи должны содержать:

* уровень;
* время;
* идентификатор запроса;
* сообщение.

---

# 12. Работа с конфигурацией

Все параметры окружения читаются через централизованный модуль конфигурации.

Запрещается:

* жёстко кодировать пути;
* хранить секреты в исходном коде.

---

# 13. Работа с базой данных

Все обращения к PostgreSQL выполняются через Repository и Unit of Work.

Запрещается:

* SQL внутри Domain Layer;
* SQL внутри API Layer;
* прямое использование ORM вне Infrastructure Layer.

---

# 14. Асинхронность

Используются `async`/`await` для:

* HTTP-запросов;
* работы с БД;
* Redis;
* WebSocket;
* файлового хранилища.

Не допускается блокирующий ввод-вывод в асинхронном коде.

---

# 15. Работа с Git

Используется Git Flow.

Основные ветки:

```text
main
develop
feature/*
bugfix/*
release/*
hotfix/*
```

Каждое изменение оформляется через Pull Request.

---

# 16. Code Review

Каждый Pull Request проверяется минимум одним разработчиком.

Проверяется:

* соответствие архитектуре;
* читаемость;
* тесты;
* типизация;
* безопасность;
* отсутствие дублирования.

---

# 17. Формат коммитов

Используется Conventional Commits.

Примеры:

```text
feat(profile): add profile search

fix(chat): resolve websocket reconnect issue

refactor(payment): simplify gateway integration

test(events): add integration tests

docs(api): update OpenAPI examples
```

---

# 18. Статический анализ

Перед каждым коммитом выполняются:

* Ruff;
* mypy;
* Bandit;
* pytest.

Коммит с ошибками не допускается.

---

# 19. Обработка ошибок

Ошибки должны:

* быть информативными;
* не раскрывать внутреннюю структуру системы;
* логироваться централизованно.

Пользователь получает только безопасные сообщения.

---

# 20. Документирование API

Все публичные эндпоинты документируются через OpenAPI.

Каждый маршрут содержит:

* описание;
* параметры;
* коды ответов;
* примеры запросов и ответов.

---

# 21. Работа с зависимостями

Перед добавлением новой библиотеки необходимо:

1. оценить необходимость;
2. проверить лицензию;
3. проверить активность поддержки;
4. проверить наличие известных уязвимостей.

Минимизируется количество внешних зависимостей.

---

# 22. Требования к производительности

Запрещается:

* выполнять N+1 запросы;
* загружать лишние данные;
* создавать неоптимальные циклы;
* выполнять тяжёлые вычисления в обработчиках HTTP.

Используются индексы, кеширование и асинхронные фоновые задачи.

---

# 23. Требования к безопасности

Запрещается:

* хранить секреты в Git;
* использовать небезопасные алгоритмы;
* отключать проверку прав доступа;
* логировать пароли, OTP и JWT.

Все пользовательские данные проходят валидацию.

---

# 24. Автоматический контроль качества

Перед объединением изменений автоматически выполняются:

1. форматирование кода;
2. сортировка импортов;
3. статический анализ;
4. проверка типов;
5. Unit Tests;
6. Integration Tests;
7. анализ безопасности;
8. проверка покрытия кода.

При любой ошибке слияние запрещается.

---

# 25. Чек-лист разработчика

Перед созданием Pull Request необходимо убедиться, что:

* архитектура не нарушена;
* код типизирован;
* тесты написаны;
* линтеры пройдены;
* документация обновлена;
* миграции созданы (при необходимости);
* отсутствуют временные комментарии (`TODO`, `FIXME`, `print()`).
