# GLOSSARY.md

# Глоссарий терминов DanceMate

**Проект:** DanceMate
**Версия:** 1.0

---

## 1. Назначение документа

Документ определяет единую терминологию проекта DanceMate.

Цели:
- единое понимание терминов всеми участниками команды;
- устранение двусмысленности в коммуникации;
- стандартизация命名ования в коде, документации и бизнес-процессах;
- упрощение онбординга новых разработчиков.

Все участники проекта обязаны использовать термины строго в соответствии с данным документом.

## 2. Архитектурные термины

### Clean Architecture
Архитектурный подход, при котором бизнес-логика (Domain) не зависит от внешних факторов (базы данных, фреймворки, API). Зависимости направлены внутрь: Infrastructure → Application → Domain.

### DDD (Domain-Driven Design)
Подход к проектированию сложных систем, основанный на моделировании предметной области. В центре — бизнес-логика и язык предметной области (Ubiquitous Language).

### Bounded Context
Граница контекста в DDD, внутри которой термины и модели имеют чёткое значение. За пределами контекста те же термины могут означать другое. Пример: "User" в контексте аутентификации и "User" в контексте профиля — это разные контексты.

### Aggregate (Агрегат)
Группа связанных объектов (Entities и Value Objects), которые рассматриваются как единое целое. У агрегата есть корневой элемент (Aggregate Root), через который осуществляется доступ ко всем внутренним объектам.

### Aggregate Root (Корень агрегата)
Единственный объект агрегата, через который внешний мир может взаимодействовать с агрегатом. Пример: `User` — корень агрегата, а `UserSettings` — внутренняя сущность, доступная только через `User`.

### Entity (Сущность)
Объект, имеющий уникальную идентичность (ID) и жизненный цикл. Может изменяться со временем. Пример: `User`, `Event`, `Invitation`.

### Value Object (Объект-значение)
Объект, который не имеет идентичности и определяется только своими атрибутами. Является неизменяемым (immutable). Пример: `Email`, `PhoneNumber`, `GeoPoint`, `Money`.

### Domain Service (Сервис предметной области)
Содержит бизнес-логику, которая не может быть естественно размещена внутри Entity или Value Object. Не имеет состояния. Пример: `MatchingService`, `RatingCalculationService`.

### Domain Event (Событие предметной области)
Уведомление о том, что в системе произошло что-то важное с точки зрения бизнеса. Генерируется агрегатом. Пример: `UserRegistered`, `InvitationAccepted`, `MeetingStarted`.

### Repository (Репозиторий)
Абстракция над хранилищем данных для агрегатов. Предоставляет методы для поиска, сохранения и удаления агрегатов. Репозиторий работает только с Aggregate Root. Интерфейс репозитория объявляется в Domain Layer, реализуется в Infrastructure Layer.

### Unit of Work (UoW)
Шаблон, обеспечивающий атомарность операций. Группирует несколько изменений в одну транзакцию. Если одна операция завершается ошибкой, откатываются все.

### Specification (Спецификация)
Шаблон для инкапсуляции бизнес-правил, которые используются для фильтрации или проверки объектов. Возвращает `true` или `false`. Пример: `CanSendInvitation`, `IsPremiumUser`.

### Factory (Фабрика)
Шаблон для создания сложных объектов или агрегатов. Инкапсулирует логику создания. Пример: `ProfileFactory`, `EventFactory`.

## 3. Слои архитектуры

### API Layer (Слой API)
Внешний интерфейс системы. Принимает HTTP и WebSocket запросы, валидирует входные данные, преобразует их в DTO, передаёт в Application Layer и возвращает ответ клиенту. Не содержит бизнес-логики.

### Application Layer (Прикладной слой)
Оркестрирует выполнение бизнес-сценариев (Use Cases). Координирует работу Domain Layer, управляет транзакциями, преобразует DTO, публикует Domain Events. Не содержит бизнес-правил — они находятся в Domain Layer.

### Domain Layer (Слой предметной области)
Ядро системы. Содержит Entities, Value Objects, Aggregates, Domain Services, Domain Events и бизнес-правила. Не зависит ни от каких других слоёв.

### Infrastructure Layer (Инфраструктурный слой)
Реализует все технические механизмы: работу с PostgreSQL, Redis, MinIO, внешними API, отправку email/SMS, очереди задач и т.д. Реализует интерфейсы, объявленные в Domain и Application слоях.

### DTO (Data Transfer Object)
Объект для передачи данных между слоями или через сеть. Не содержит бизнес-логики. Используется для сериализации/десериализации.

### Use Case (Сценарий использования)
Конкретная бизнес-операция, которую может выполнить пользователь. Пример: `CreateProfileUseCase`, `SendInvitationUseCase`.

### CQRS (Command Query Responsibility Segregation)
Шаблон, разделяющий операции на команды (изменение состояния) и запросы (чтение). Команды и запросы имеют разные обработчики и, возможно, разные модели данных.

### Command (Команда)
Запрос на изменение состояния системы. Не возвращает данные (кроме идентификатора или статуса). Пример: `CreateProfileCommand`, `AcceptInvitationCommand`.

### Query (Запрос)
Запрос на чтение данных. Не изменяет состояние системы. Возвращает данные. Пример: `GetProfileQuery`, `SearchProfilesQuery`.

### Event Handler (Обработчик события)
Компонент, который реагирует на Domain Event. Может отправлять уведомления, обновлять статистику, запускать фоновые задачи. Располагается в Application Layer.

## 4. Бизнес-термины (Domain)

### Account (Аккаунт)
Учётная запись пользователя. Содержит телефон, пароль (хеш), статус верификации. Отвечает за аутентификацию и авторизацию.

### User (Пользователь)
Расширенная модель пользователя. Содержит роли, настройки, блокировки, избранные города. Связан с Account отношением 1:1.

### Profile (Профиль)
Анкета танцора. Содержит имя, возраст, описание, фотографии, видео, танцевальные стили, уровень, предпочтения, город. Является лицом пользователя в сообществе.

### Dance Style (Танцевальный стиль)
Направление танца. Примеры: Salsa, Bachata, Tango, Кизомба, WCS, Хастл. Используется как справочник и для поиска партнёров.

### Dance Level (Уровень танца)
Уровень мастерства в конкретном стиле. Примеры: Beginner, Intermediate, Advanced, Professional.

### Invitation (Приглашение)
Предложение одному пользователю потанцевать с другим. Содержит отправителя, получателя, статус (Pending, Accepted, Declined, Expired), комментарий, время.

### Meeting (Встреча)
Факт совместной тренировки или танца. Создаётся после принятия приглашения. Содержит участников, время, место, результат.

### Event (Событие / Мероприятие)
Танцевальное событие: фестиваль, вечеринка, мастер-класс, тренировка. Содержит создателя, название, описание, дату, место, участников.

### Squad (Сквад / Группа)
Групповое объединение пользователей для совместных поездок на мероприятия. Содержит название, организатора, участников, цели.

### Chat (Чат)
Диалог между двумя или более пользователями. Может быть личным (private) или групповым (group).

### Message (Сообщение)
Отдельное сообщение в чате. Содержит отправителя, текст, вложения, время отправки.

### Rating (Рейтинг / Оценка)
Оценка, которую пользователь ставит другому пользователю после встречи. Значение от 1 до 5. Влияет на репутацию.

### Gratitude (Благодарность)
Публичная благодарность пользователя другому пользователю. Отображается в профиле. Является позитивным сигналом для сообщества.

### Notification (Уведомление)
Сообщение для пользователя о событиях в системе: новые приглашения, сообщения, изменения статуса событий, напоминания.

### Subscription (Подписка)
Платная подписка, открывающая доступ к дополнительным функциям: расширенный поиск, неограниченные приглашения, приоритетная поддержка.

### Payment (Платёж)
Финансовая операция по оплате подписки. Содержит сумму, статус, способ оплаты, ссылку на подписку.

### Moderation (Модерация)
Процесс проверки контента и поведения пользователей. Включает жалобы, блокировки, предупреждения.

### Report (Жалоба)
Сообщение от пользователя о нарушении правил другим пользователем или контентом.

### Ban (Блокировка)
Ограничение доступа пользователя к системе или её функциям. Может быть временной или постоянной.

### Achievement (Достижение)
Награда пользователю за определённые действия: регистрация, первая встреча, 10 встреч, получение благодарности и т.д.

## 5. Технические термины

### PostgreSQL
Основная реляционная база данных. Источник истины (source of truth) для всех данных.

### Redis
In-memory хранилище. Используется для кеширования, хранения сессий, rate limiting, очередей задач и pub/sub.

### MinIO
Объектное хранилище, совместимое с S3 API. Используется для хранения медиафайлов: фотографий, видео, документов.

### SQLAlchemy
ORM (Object-Relational Mapper) для работы с PostgreSQL в Python. Используется только в Infrastructure Layer.

### Alembic
Инструмент для управления миграциями базы данных. Все изменения схемы PostgreSQL выполняются через Alembic.

### FastAPI
Web-фреймворк для создания REST API и WebSocket на Python. Используется для реализации API Layer.

### JWT (JSON Web Token)
Токен для аутентификации пользователей. Содержит информацию о пользователе и его правах. Имеет ограниченный срок жизни (Access Token — 15 минут, Refresh Token — 30 дней).

### OTP (One-Time Password)
Одноразовый пароль, отправляемый по SMS для верификации номера телефона при регистрации и входе. Действует 5 минут.

### RBAC (Role-Based Access Control)
Модель управления доступом, основанная на ролях. Каждая роль имеет набор разрешений (permissions).

### Rate Limiting
Ограничение количества запросов от одного пользователя или IP за определённое время. Защищает систему от злоупотреблений.

### WebSocket
Протокол для двусторонней связи в реальном времени. Используется для чатов, уведомлений и статуса пользователей.

### API Gateway
Единая точка входа для всех клиентов. Маршрутизирует запросы к соответствующим сервисам. (Планируется на будущих этапах).

### Materialized View
Материализованное представление в PostgreSQL. Хранит результаты сложного запроса для ускорения чтения. Используется для аналитики и рекомендаций.

### Soft Delete
Подход, при котором данные не удаляются физически, а помечаются как удалённые (поле `deleted_at`). Используется для большинства бизнес-сущностей.

### Migration (Миграция)
Изменение схемы базы данных, выполняемое через Alembic. Каждая миграция является версионированной и воспроизводимой.

### Health Check
Эндпоинт для проверки состояния сервиса. Используется для мониторинга и автоматического перезапуска.

### Observability (Наблюдаемость)
Способность системы предоставлять информацию о своём состоянии через метрики, логи и трассировку.

### Tracing (Трассировка)
Отслеживание пути запроса через все сервисы системы. Используется для анализа производительности и поиска узких мест.

## 6. Метрики и показатели

### DAU (Daily Active Users)
Количество уникальных пользователей, которые взаимодействовали с системой за день.

### MAU (Monthly Active Users)
Количество уникальных пользователей, которые взаимодействовали с системой за месяц.

### Retention (Удержание)
Процент пользователей, вернувшихся в систему через определённое время после регистрации (D1, D7, D30).

### P95 / P99 (Перцентили)
Показатели времени ответа: 95% и 99% запросов выполняются быстрее указанного значения.

### SLA (Service Level Agreement)
Соглашение об уровне обслуживания. Определяет гарантируемые показатели доступности и производительности.

### SLO (Service Level Objective)
Целевой показатель уровня обслуживания. Конкретная метрика, которую система должна достигать (например, время ответа ≤ 300 мс).

### SLI (Service Level Indicator)
Фактический показатель уровня обслуживания. Реальное измеряемое значение метрики.

### MTTR (Mean Time To Recovery)
Среднее время восстановления системы после сбоя.

## 7. Правила использования терминов

1. Все термины из этого глоссария являются обязательными для использования в коде, документации и устной коммуникации.
2. Запрещается использовать синонимы для терминов, определённых в глоссарии. Пример: "встреча" и "рандеву" — только `Meeting`.
3. При добавлении нового термина он должен быть сначала включён в глоссарий, а затем использован в коде.
4. Англоязычные термины (`User`, `Profile`, `Event`) используются в коде и технической документации. В пользовательской документации и UI используются русскоязычные эквиваленты.

---

## 8. Изменение глоссария

Изменения в глоссарий вносятся через Pull Request с обязательным Code Review. При изменении термина необходимо:

1. Обновить определение в глоссарии.
2. Проверить все документы на предмет использования устаревшего термина.
3. Обновить код, если терминология изменилась (например, переименование класса).
4. Уведомить команду об изменении.
