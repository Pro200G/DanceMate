# Performance Guide

**Проект:** DanceMate

**Версия:** 1.0

---

# 1. Назначение

Документ определяет стратегию обеспечения производительности приложения DanceMate.

Основные цели:

* минимальное время отклика;
* эффективное использование ресурсов;
* горизонтальное масштабирование;
* снижение нагрузки на PostgreSQL;
* оптимизация работы с сетью;
* поддержка высокой конкурентной нагрузки.

---

# 2. Целевые показатели производительности

| Метрика                  | Целевое значение |
| ------------------------ | ---------------: |
| Среднее время ответа API |         ≤ 300 мс |
| P95 API                  |         ≤ 700 мс |
| P99 API                  |        ≤ 1000 мс |
| Создание профиля         |         ≤ 500 мс |
| Поиск пользователей      |         ≤ 200 мс |
| Создание мероприятия     |         ≤ 500 мс |
| Отправка сообщения       |         ≤ 150 мс |
| WebSocket задержка       |         ≤ 100 мс |
| Время запуска Backend    |         ≤ 10 сек |

---

# 3. Архитектурные принципы

Производительность обеспечивается за счёт:

* асинхронной архитектуры;
* кеширования;
* оптимизированных SQL-запросов;
* индексации;
* материализованных представлений;
* фоновой обработки тяжёлых операций;
* минимизации сетевых запросов.

---

# 4. Производительность PostgreSQL

Основные требования:

* использование индексов для всех часто используемых условий поиска;
* исключение последовательных сканирований больших таблиц без необходимости;
* регулярный `VACUUM` и `ANALYZE`;
* контроль блокировок и длительных транзакций;
* применение `EXPLAIN ANALYZE` при оптимизации запросов.

---

# 5. Индексация

Используются:

* B-tree;
* GIN;
* GiST;
* Partial Index;
* Composite Index;
* Covering Index (INCLUDE).

Каждый новый запрос анализируется на предмет необходимости дополнительного индекса.

---

# 6. Materialized Views

Используются для:

* рекомендаций пользователей;
* популярности мероприятий;
* статистики;
* аналитических отчётов;
* рейтингов.

Обновление выполняется фоновыми задачами по расписанию или после существенных изменений данных.

---

# 7. SQLAlchemy

Требования:

* использование SQLAlchemy 2.x;
* асинхронный драйвер `asyncpg`;
* применение `selectinload()` и `joinedload()` для предотвращения проблемы N+1;
* выбор только необходимых столбцов (`load_only`, `defer`);
* пагинация для коллекций.

Запрещается выполнять ленивую загрузку больших коллекций в циклах.

---

# 8. Кеширование

Используется Redis.

Кешируются:

* профили пользователей;
* список танцевальных стилей;
* настройки системы;
* результаты популярных поисковых запросов;
* временные токены.

Политика кеширования:

* TTL для изменяемых данных;
* инвалидация при изменении источника данных.

---

# 9. WebSocket

Оптимизация включает:

* минимальный размер сообщений;
* пакетную отправку при необходимости;
* контроль количества подключений;
* ограничение частоты сообщений;
* периодические Ping/Pong для проверки соединения.

---

# 10. Фоновые задачи

Тяжёлые операции выполняются вне HTTP-запросов:

* отправка уведомлений;
* генерация отчётов;
* обработка изображений;
* обновление Materialized Views;
* очистка устаревших данных.

Используется очередь задач (например, Celery или Dramatiq).

---

# 11. Работа с медиа

Изображения:

* автоматическое изменение размера;
* генерация миниатюр;
* оптимизация качества.

Видео:

* транскодирование;
* создание превью;
* хранение в MinIO.

Медиафайлы не обрабатываются синхронно в HTTP-запросах.

---

# 12. HTTP API

Рекомендуется:

* сжатие ответов (gzip или brotli);
* использование HTTP Keep-Alive;
* корректные заголовки кеширования;
* ETag и Last-Modified для неизменяемых ресурсов;
* минимизация размера JSON.

---

# 13. Оптимизация сети

Используются:

* HTTP/2 или HTTP/3 (при поддержке инфраструктуры);
* TLS 1.3;
* постоянные соединения;
* CDN для статических ресурсов (при необходимости).

---

# 14. Масштабирование

Поддерживаются:

* горизонтальное масштабирование Backend;
* масштабирование Worker-процессов;
* увеличение ресурсов PostgreSQL;
* масштабирование Redis;
* масштабирование MinIO.

Все экземпляры Backend являются статeless.

---

# 15. Нагрузочное тестирование

Проверяются сценарии:

* массовая регистрация;
* поиск пользователей;
* обмен сообщениями;
* создание мероприятий;
* загрузка медиа;
* работа WebSocket.

Инструменты:

* k6;
* Locust.

---

# 16. Профилирование

Для анализа производительности используются:

* `cProfile`;
* `py-spy`;
* OpenTelemetry Tracing;
* `EXPLAIN ANALYZE` для SQL;
* метрики Prometheus.

Профилирование проводится регулярно для выявления узких мест.

---

# 17. Мониторинг производительности

Контролируются:

* время ответа API;
* использование CPU;
* использование памяти;
* количество активных подключений;
* длительность SQL-запросов;
* размер очередей задач;
* задержка WebSocket.

---

# 18. Работа с памятью

Требования:

* отсутствие утечек памяти;
* ограничение размеров кеша;
* освобождение неиспользуемых ресурсов;
* контроль потребления памяти контейнерами.

---

# 19. Оптимизация Frontend-взаимодействия

Backend поддерживает:

* пагинацию;
* фильтрацию;
* сортировку;
* выбор полей ответа (при необходимости);
* пакетную обработку запросов.

Это снижает объём передаваемых данных.

---

# 20. Производительность базы данных

Рекомендуемые параметры:

* пул соединений SQLAlchemy;
* ограничение максимального числа подключений;
* настройка `work_mem`, `shared_buffers`, `effective_cache_size` в PostgreSQL;
* регулярный анализ медленных запросов.

Настройки должны корректироваться по результатам мониторинга.

---

# 21. Оптимизация запросов

Запрещается:

* `SELECT *` без необходимости;
* длительные транзакции;
* N+1 запросы;
* отсутствие пагинации для больших выборок;
* выполнение тяжёлых вычислений в SQL без обоснования.

---

# 22. Регламент оптимизации

При обнаружении деградации производительности выполняются:

1. Анализ метрик.
2. Анализ трассировки.
3. Анализ SQL.
4. Оптимизация индексов.
5. Проверка кеширования.
6. Повторное нагрузочное тестирование.
7. Документирование изменений.

---

# 23. Чек-лист перед релизом

Перед выпуском новой версии необходимо убедиться, что:

* отсутствуют медленные SQL-запросы;
* все критичные запросы используют индексы;
* кеширование настроено корректно;
* нагрузочные тесты пройдены;
* показатели соответствуют установленным целям;
* мониторинг и алерты настроены.

---
