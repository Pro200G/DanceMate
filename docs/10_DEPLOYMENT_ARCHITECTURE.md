# 09_DEPLOYMENT_ARCHITECTURE.md

# Deployment Architecture

**Проект:** DanceMate
**Версия:** 1.0

---

# 1. Назначение

Документ описывает архитектуру развёртывания приложения DanceMate.

Определяются:

* окружения разработки и эксплуатации;
* контейнеризация;
* сетевая архитектура;
* порядок запуска сервисов;
* масштабирование;
* CI/CD;
* резервное копирование;
* мониторинг;
* стратегия обновления.

Deployment Architecture обеспечивает воспроизводимое и безопасное развертывание приложения.

---

# 2. Архитектура развертывания

```text
                        Internet
                            │
                     HTTPS (TLS 1.3)
                            │
                       Nginx Reverse Proxy
                            │
          ┌─────────────────┴─────────────────┐
          │                                   │
     FastAPI Backend                    Static Frontend
          │
          ├──────────────┐
          │              │
      PostgreSQL       Redis
          │              │
          └──────────────┘
                 │
               MinIO
                 │
           Background Workers
```

---

# 3. Среды выполнения

Поддерживаются четыре окружения:

### Development

Используется разработчиками.

Особенности:

* Docker Compose;
* Debug Mode;
* тестовые данные;
* локальный Redis;
* локальный PostgreSQL.

---

### Testing

Используется автоматическими тестами.

Особенности:

* изолированная БД;
* автоматическое создание схемы;
* фикстуры;
* очистка после тестов.

---

### Staging

Используется перед выпуском новой версии.

Максимально приближено к Production.

---

### Production

Боевая среда.

Особенности:

* HTTPS;
* резервное копирование;
* мониторинг;
* централизованное логирование;
* автоматический перезапуск сервисов.

---

# 4. Docker

Каждый компонент запускается в собственном контейнере.

```text
docker/
├── backend/
├── nginx/
├── postgres/
├── redis/
├── minio/
└── monitoring/
```

Контейнеры изолированы друг от друга.

---

# 5. Docker Compose

Локальная разработка выполняется через Docker Compose.

Основные сервисы:

* backend;
* postgres;
* redis;
* minio;
* nginx;
* worker.

Все сервисы запускаются одной командой.

---

# 6. Сетевая архитектура

Используются отдельные Docker-сети:

```text
frontend_network
backend_network
database_network
monitoring_network
```

Доступ к PostgreSQL и Redis извне запрещён.

---

# 7. PostgreSQL

Используется отдельный контейнер PostgreSQL.

Требования:

* ежедневное резервное копирование;
* WAL;
* регулярный VACUUM;
* контроль размеров таблиц;
* мониторинг производительности.

---

# 8. Redis

Redis используется исключительно для:

* кеширования;
* хранения сессий (при необходимости);
* очередей задач;
* ограничения частоты запросов.

Redis не является постоянным хранилищем данных.

---

# 9. MinIO

MinIO используется для хранения:

* фотографий;
* видео;
* документов;
* медиафайлов мероприятий.

Все объекты доступны только по временным подписанным ссылкам.

---

# 10. Background Workers

Фоновые процессы выполняются отдельно от API.

Основные задачи:

* отправка уведомлений;
* обработка изображений;
* очистка временных данных;
* обновление Materialized Views;
* формирование статистики.

---

# 11. Nginx

Nginx выполняет функции:

* TLS Termination;
* Reverse Proxy;
* сжатие ответов;
* кеширование статических файлов;
* ограничение количества запросов;
* маршрутизация.

---

# 12. Масштабирование

Backend масштабируется горизонтально.

```text
          Nginx
        /   |   \
API-1  API-2  API-3
```

Все экземпляры используют:

* PostgreSQL;
* Redis;
* MinIO.

---

# 13. Конфигурация

Конфигурация хранится через:

* `.env`;
* Docker Secrets (Production);
* переменные окружения.

Запрещается хранить секреты в исходном коде.

---

# 14. Логирование

Все сервисы отправляют логи в единое хранилище.

Логируются:

* HTTP-запросы;
* ошибки;
* фоновые задачи;
* обращения к БД;
* события безопасности.

---

# 15. Мониторинг

Контролируются:

* использование CPU;
* память;
* дисковое пространство;
* время ответа API;
* количество соединений PostgreSQL;
* использование Redis;
* очереди фоновых задач.

---

# 16. Health Checks

Каждый сервис реализует проверку состояния.

Проверяются:

* API;
* PostgreSQL;
* Redis;
* MinIO;
* Workers.

Nginx использует эти проверки для маршрутизации запросов.

---

# 17. CI/CD

Процесс поставки включает:

1. Проверку кода.
2. Запуск тестов.
3. Статический анализ.
4. Проверку зависимостей.
5. Сборку Docker-образов.
6. Создание миграций.
7. Развёртывание в Staging.
8. Автоматические проверки.
9. Развёртывание в Production.

---

# 18. Обновление базы данных

Все изменения схемы выполняются только через Alembic.

Порядок:

1. Создание миграции.
2. Проверка.
3. Резервное копирование.
4. Выполнение миграции.
5. Проверка целостности.

---

# 19. Резервное копирование

Создаются:

* ежедневные резервные копии PostgreSQL;
* еженедельные полные архивы;
* резервные копии MinIO;
* экспорт конфигурации.

Периодически проводится тестовое восстановление.

---

# 20. Стратегия восстановления

При сбое:

1. Остановка приложения.
2. Восстановление PostgreSQL.
3. Восстановление MinIO.
4. Проверка миграций.
5. Запуск Backend.
6. Проверка API.
7. Возврат пользователей.

---

# 21. Масштабирование проекта

Архитектура допускает переход к Kubernetes без изменения структуры приложения.

Каждый сервис является независимым контейнером.

---

# 22. Требования к Production

Обязательные требования:

* HTTPS;
* автоматическое обновление сертификатов;
* ежедневные резервные копии;
* мониторинг;
* централизованные логи;
* ограничение доступа к инфраструктуре;
* регулярное обновление зависимостей.
