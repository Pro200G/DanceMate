# 16_MONITORING.md

# Monitoring System

**Проект:** DanceMate
**Версия:** 1.0

---

# 1. Назначение

Документ описывает практическую реализацию системы мониторинга проекта DanceMate.

Цели:

* контроль состояния инфраструктуры;
* контроль состояния приложений;
* раннее обнаружение инцидентов;
* визуализация метрик;
* автоматическое оповещение;
* поддержка SLA/SLO.

---

# 3. Основные компоненты

## 3.1 Prometheus

Используется для:

* сбора метрик;
* хранения временных рядов;
* выполнения alert-правил.

---

## 3.2 Grafana

Используется для:

* визуализации метрик;
* построения дашбордов;
* анализа производительности;
* расследования инцидентов.

---

## 3.3 Loki

Используется для:

* централизованного хранения логов;
* поиска логов по запросам;
* корреляции логов с метриками.

---

## 3.4 OpenTelemetry

Используется для:

* распределённой трассировки;
* сбора trace/span данных;
* передачи данных в Grafana Tempo или аналог.

---

## 3.5 Alertmanager

Используется для:

* обработки алертов;
* дедупликации уведомлений;
* маршрутизации инцидентов.

---

# 4. Источники метрик

Метрики собираются из:

* FastAPI backend;
* PostgreSQL;
* Redis;
* MinIO;
* Docker контейнеров;
* Background Workers;
* Nginx.

---

# 5. Типы метрик

## 5.1 Application Metrics

* количество запросов;
* latency (p50, p95, p99);
* количество ошибок;
* активные пользователи;
* WebSocket соединения.

---

## 5.2 Database Metrics

* количество подключений;
* медленные запросы;
* блокировки;
* использование индексов;
* размер таблиц;
* cache hit ratio.

---

## 5.3 Redis Metrics

* memory usage;
* key eviction rate;
* hit/miss ratio;
* latency;
* количество соединений.

---

## 5.4 System Metrics

* CPU usage;
* RAM usage;
* disk usage;
* network I/O;
* container restarts.

---

## 5.5 Business Metrics

* регистрация пользователей;
* создание профилей;
* активные события;
* сообщения в чатах;
* транзакции;
* загрузки медиа.

---

# 6. Дашборды Grafana

## 6.1 Основные дашборды

* API Overview
* System Health
* Database Performance
* Redis Performance
* WebSocket Activity
* Background Jobs
* Business Analytics

---

## 6.2 Структура дашбордов

Каждый дашборд содержит:

* общие метрики;
* графики времени ответа;
* ошибки;
* нагрузку;
* алерты.

---

# 7. Алерты

## 7.1 Критические алерты

* недоступность API;
* падение PostgreSQL;
* недоступность Redis;
* переполнение диска;
* рост 5xx ошибок;
* падение worker-ов.

---

## 7.2 Предупреждения

* рост latency;
* увеличение очередей задач;
* снижение cache hit ratio;
* рост количества медленных запросов.

---

## 7.3 Информационные

* деплой новой версии;
* масштабирование сервисов;
* обновление конфигураций.

---

# 8. SLI / SLO

## 8.1 SLI (Service Level Indicators)

* latency;
* availability;
* error rate;
* throughput.

---

## 8.2 SLO (Service Level Objectives)

| Метрика          |     Цель |
| ---------------- | -------: |
| Uptime API       |  ≥ 99.9% |
| Error Rate       |     ≤ 1% |
| P95 Latency      | ≤ 700 ms |
| WebSocket uptime |  ≥ 99.5% |

---

## 8.3 SLA (Service Level Agreement)

В случае нарушения:

* автоматический инцидент;
* анализ причин;
* postmortem.

---

# 9. Логирование

## 9.1 Структура логов

Все логи в формате JSON:

```json id="log1"
{
  "timestamp": "2026-06-27T10:00:00Z",
  "level": "INFO",
  "service": "backend",
  "request_id": "abc123",
  "user_id": "uuid",
  "message": "User created profile"
}
```

---

## 9.2 Типы логов

* application logs;
* access logs;
* security logs;
* audit logs;
* system logs;
* worker logs.

---

## 9.3 Хранение логов

| Тип         | Срок хранения |
| ----------- | ------------: |
| Application |       30 дней |
| Access      |       30 дней |
| Security    |         1 год |
| Audit       |         1 год |

---

# 10. Трассировка

## 10.1 Trace Context

Каждый запрос содержит:

* trace_id;
* span_id;
* parent_span_id.

---

## 10.2 Границы трассировки

Отслеживаются:

* HTTP запросы;
* SQL запросы;
* Redis операции;
* внешние API;
* background tasks.

---

# 11. Health Checks

## 11.1 Endpoints

```text id="health"
GET /health/live
GET /health/ready
GET /health/startup
```

---

## 11.2 Проверки

* API доступен;
* DB подключение;
* Redis доступен;
* MinIO доступен;
* worker-ы активны.

---

# 12. Инциденты

## 12.1 Классификация

* P0 — полный отказ системы;
* P1 — деградация критических функций;
* P2 — частичные сбои;
* P3 — незначительные проблемы.

---

## 12.2 Обработка инцидентов

1. фиксация;
2. уведомление;
3. диагностика;
4. устранение;
5. постмортем.

---

# 13. Postmortem

Каждый инцидент документируется:

* причина;
* влияние;
* таймлайн;
* решение;
* профилактика.

---

# 14. Уведомления

Каналы:

* Telegram;

# 15. Мониторинг CI/CD

# 17. Эксплуатационные правила

* мониторинг обязателен для всех сервисов;
* отсутствие метрик запрещено;
* каждый новый endpoint должен иметь метрики;
* все ошибки должны логироваться;
* все инциденты должны фиксироваться.
