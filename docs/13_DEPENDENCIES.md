# 13_DEPENDENCIES.md

# Dependencies

**Проект:** DanceMate

**Версия:** 1.0

---

# 1. Назначение

Документ определяет политику использования внешних зависимостей проекта.

Цели:

* единый стек технологий;
* контроль версий библиотек;
* безопасность зависимостей;
* воспроизводимые сборки;
* совместимость компонентов.

Все сторонние библиотеки должны быть утверждены архитектурной документацией.

---

# 2. Принципы выбора зависимостей

При выборе новой библиотеки учитываются:

* активная поддержка;
* совместимость с Python 3.13;
* совместимость с PostgreSQL 17;
* совместимость с FastAPI;
* лицензия (MIT, BSD, Apache 2.0);
* отсутствие известных критических уязвимостей;
* регулярные обновления.

Предпочтение отдаётся небольшому количеству зрелых библиотек.

---

# 3. Основной стек технологий

| Компонент          | Технология     |
| ------------------ | -------------- |
| Язык               | Python 3.13    |
| Backend            | FastAPI        |
| ORM                | SQLAlchemy 2.x |
| Миграции           | Alembic        |
| База данных        | PostgreSQL 17  |
| Валидация          | Pydantic v2    |
| Кэш                | Redis          |
| Файловое хранилище | MinIO (S3 API) |
| ASGI Server        | Uvicorn        |
| Контейнеризация    | Docker         |
| Reverse Proxy      | Nginx          |

---

# 4. Backend Framework

Используется FastAPI.

Основные причины выбора:

* высокая производительность;
* асинхронная архитектура;
* OpenAPI 3.1;
* автоматическая генерация документации;
* встроенная Dependency Injection.

---

# 5. Работа с базой данных

Используются:

* SQLAlchemy 2.x;
* Alembic;
* asyncpg.

SQLAlchemy используется только в Infrastructure Layer.

---

# 6. Работа с Redis

Redis применяется для:

* кеширования;
* ограничения частоты запросов;
* хранения временных данных;
* брокера сообщений (при необходимости).

Redis не используется как основное хранилище данных.

---

# 7. Работа с файлами

Используется MinIO.

Поддерживаются:

* фотографии;
* видео;
* документы;
* временные файлы.

Все объекты хранятся вне PostgreSQL.

---

# 8. Аутентификация

Используются:

* JWT;
* Refresh Token;
* OTP;
* Argon2id (при использовании паролей).

---

# 9. Асинхронность

Используются:

* asyncio;
* async/await;
* httpx;
* WebSocket.

Блокирующие операции запрещены в обработчиках HTTP.

---

# 10. Очереди задач

Фоновые задачи выполняются независимо от API.

Рекомендуемый стек:

* Celery + Redis;

или

* Dramatiq + Redis.

Выбор фиксируется на этапе реализации.

---

# 11. Тестирование

Основные библиотеки:

| Назначение       | Библиотека     |
| ---------------- | -------------- |
| Unit Tests       | pytest         |
| Async Tests      | pytest-asyncio |
| Покрытие         | pytest-cov     |
| HTTP Tests       | httpx          |
| Генерация данных | Faker          |
| Factory          | factory_boy    |

---

# 12. Проверка качества

Используются:

* Ruff;
* mypy;
* Bandit;
* pre-commit.

Все проверки выполняются автоматически.

---

# 13. Документирование

Используются:

* OpenAPI;
* Swagger UI;
* ReDoc;
* Markdown.

---

# 14. Мониторинг

Рекомендуемый стек:

* Prometheus;
* Grafana;
* OpenTelemetry.

Логирование осуществляется через стандартный модуль `logging`.

---

# 15. Работа с изображениями

Для обработки медиа используются:

* Pillow;
* ffmpeg (через отдельный сервис или контейнер).

Все тяжёлые операции выполняются в фоне.

---

# 16. Работа с платежами

Интерфейс платёжного шлюза абстрагирован.

Возможные реализации:

* Stripe;
* YooKassa;
* CloudPayments.

Замена провайдера не должна влиять на бизнес-логику.

---

# 17. Управление зависимостями

Используется:

```text
pyproject.toml
```

Для воспроизводимых сборок дополнительно фиксируется файл блокировки зависимостей (например, `uv.lock` или аналогичный для выбранного менеджера пакетов).

Запрещается устанавливать библиотеки вручную в рабочем окружении без фиксации их версий.

---

# 18. Обновление зависимостей

Перед обновлением выполняются:

1. анализ изменений;
2. проверка совместимости;
3. запуск полного набора тестов;
4. обновление документации.

Критические обновления безопасности устанавливаются в приоритетном порядке.

---

# 19. Запрещённые зависимости

Не допускается использование библиотек:

* без активной поддержки;
* с критическими уязвимостями;
* с несовместимой лицензией;
* дублирующих существующий функционал.

Каждая новая зависимость должна иметь обоснование.

---


---

# 21. Контроль безопасности

Перед каждым релизом автоматически проверяются:

* известные CVE;
* устаревшие версии;
* несовместимые обновления;
* лицензии сторонних библиотек.

Используются автоматизированные средства анализа зависимостей.
