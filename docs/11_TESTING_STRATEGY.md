# Testing Strategy

**Проект:** DanceMate
**Версия:** 1.0

---

# 1. Назначение

Документ определяет стратегию тестирования системы DanceMate.

Цели тестирования:

* подтверждение корректности бизнес-логики;
* предотвращение регрессий;
* проверка безопасности;
* контроль производительности;
* обеспечение стабильности релизов;
* автоматизация проверки качества.

Тестирование является обязательной частью процесса разработки и CI/CD.

---

# 2. Пирамида тестирования

```text
                 E2E Tests
                    ▲
             Integration Tests
                    ▲
               Component Tests
                    ▲
                Unit Tests
```

Основной объём тестов составляют Unit Tests.

# 4. Unit Tests

Unit Tests проверяют отдельные классы и функции.

Покрываются:

* Domain Entities;
* Value Objects;
* Domain Services;
* Specifications;
* Validators;
* Use Cases;
* Utility-функции.

Unit Tests не используют:

* PostgreSQL;
* Redis;
* MinIO;
* HTTP;
* WebSocket.

Все внешние зависимости заменяются mock-объектами.

---

# 5. Integration Tests

Integration Tests проверяют взаимодействие компонентов.

Тестируются:

* SQLAlchemy;
* PostgreSQL;
* Redis;
* MinIO;
* Repository;
* Unit of Work;
* очереди задач.

Используется отдельная тестовая база данных.

---

# 6. API Tests

API Tests проверяют:

* REST API;
* WebSocket API;
* авторизацию;
* валидацию;
* сериализацию;
* обработку ошибок;
* формат ответов.

Проверяются все публичные эндпоинты.

---

# 7. Contract Tests

Contract Tests гарантируют совместимость между:

* Backend ↔ Frontend;
* Backend ↔ Mobile;
* Backend ↔ внешними сервисами.

Проверяются:

* структура JSON;
* типы данных;
* обязательные поля;
* версии API.

---

# 8. Security Tests

Проверяются:

* JWT;
* Refresh Token;
* RBAC;
* права доступа;
* SQL Injection;
* XSS;
* CSRF (для Web-клиента);
* Brute Force;
* Replay Attack;
* загрузка вредоносных файлов;
* ограничения Rate Limiting.

---

# 9. Performance Tests

Измеряются:

* время ответа API;
* скорость поиска;
* скорость создания мероприятий;
* скорость отправки сообщений;
* работа WebSocket;
* обработка фоновых задач.

Целевые показатели:

* среднее время ответа API — до 300 мс;
* 95-й перцентиль — до 700 мс.

---

# 10. Load Tests

Проверяется работа системы при высокой нагрузке.

Сценарии:

* массовый вход пользователей;
* массовая отправка сообщений;
* одновременный поиск;
* загрузка медиа;
* создание мероприятий.

Используются инструменты:

* Locust;
* k6.

---

# 11. End-to-End Tests

E2E Tests проверяют полный пользовательский сценарий.

Примеры:

* регистрация пользователя;
* создание профиля;
* поиск партнёра;
* отправка приглашения;
* подтверждение приглашения;
* создание встречи;
* обмен сообщениями;
* покупка подписки.

Используются реальные API и тестовая инфраструктура.

---

# 12. Test Fixtures

Fixtures создают повторно используемые объекты.

Примеры:

```text
fixtures/
├── users.py
├── profiles.py
├── events.py
├── meetings.py
├── chats.py
└── payments.py
```

---

# 13. Test Factories

Factories генерируют тестовые данные.

```text
factories/
├── user_factory.py
├── profile_factory.py
├── event_factory.py
├── meeting_factory.py
└── message_factory.py
```

---

# 14. Mock Objects

Используются для замены:

* SMS;
* Email;
* платёжных шлюзов;
* MinIO;
* Redis;
* WebSocket;
* внешних API.

Mocks должны повторять реальные интерфейсы.

---

# 15. Тестовые данные

Отдельный каталог:

```text
data/
├── users.json
├── events.json
├── profiles.json
├── messages.json
└── media/
```

Данные не должны содержать персональную информацию.

---

# 16. Покрытие кода

Минимальное покрытие:

| Компонент            | Покрытие |
| -------------------- | -------: |
| Domain Layer         |    ≥ 95% |
| Application Layer    |    ≥ 90% |
| Infrastructure Layer |    ≥ 80% |
| API Layer            |    ≥ 85% |
| Общий проект         |    ≥ 90% |

---

# 17. Автоматический запуск

Тесты выполняются:

* перед каждым Pull Request;
* перед слиянием в основную ветку;
* перед выпуском релиза;
* по расписанию (ночной прогон полного набора тестов).

---

# 18. CI/CD Pipeline

Каждый коммит проходит:

1. Проверку форматирования.
2. Статический анализ.
3. Проверку типов.
4. Unit Tests.
5. Integration Tests.
6. API Tests.
7. Security Tests.
8. Сборку Docker-образов.
9. Проверку миграций.
10. Развёртывание в Staging.

При любой ошибке конвейер останавливается.

---

# 19. Производительность тестов

Целевые показатели:

* Unit Tests — до 5 минут;
* Integration Tests — до 15 минут;
* Полный набор тестов — до 30 минут.

Длительные проверки (нагрузочные, E2E) запускаются отдельно.

---

# 20. Правила разработки тестов

Для каждого нового функционала обязательно создаются:

* Unit Tests;
* Integration Tests (если есть взаимодействие с инфраструктурой);
* API Tests (для публичных эндпоинтов).

Исправление ошибки должно сопровождаться тестом, воспроизводящим найденный дефект.

---

# 21. Метрики качества

Контролируются:

* процент покрытия кода;
* количество упавших тестов;
* время выполнения тестов;
* количество дефектов после релиза;
* частота регрессий.

Эти показатели отслеживаются автоматически.

---

# 22. Инструменты

Основные инструменты:

* pytest;
* pytest-asyncio;
* pytest-cov;
* httpx (для тестирования API);
* factory_boy;
* Faker;
* Locust;
* k6.

Допускается использование дополнительных библиотек при необходимости.

---

# 23. Правила именования

Имена файлов:

```text
test_profile_service.py
test_create_event_use_case.py
test_payment_repository.py
```

Имена тестов:

```python
def test_user_can_create_profile():
    ...

def test_invitation_expires_after_timeout():
    ...

def test_subscription_cannot_be_used_after_expiration():
    ...
```

Название теста должно описывать ожидаемое поведение.

---

# 24. Требования к релизу

Релиз допускается только при выполнении следующих условий:

* все тесты успешно пройдены;
* покрытие соответствует установленным требованиям;
* отсутствуют критические ошибки безопасности;
* миграции успешно применяются;
* успешно пройдены интеграционные проверки.
