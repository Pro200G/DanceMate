# Security Architecture

**Проект:** DanceMate
**Версия:** 1.0

---

# 1. Назначение

Документ определяет архитектуру безопасности приложения DanceMate.

Цели:

* защита персональных данных пользователей;
* обеспечение конфиденциальности сообщений;
* защита платёжных операций;
* защита REST API и WebSocket;
* контроль доступа;
* аудит действий;
* минимизация последствий компрометации отдельных компонентов.

---

# 2. Принципы безопасности

Система строится на следующих принципах:

* Zero Trust;
* Least Privilege;
* Defense in Depth;
* Secure by Default;
* Fail Secure;
* Principle of Separation of Duties.

---

# 3. Архитектура безопасности

```text
Client
   │
HTTPS / TLS 1.3
   │
NGINX
   │
Rate Limiter
   │
FastAPI
   │
Authentication
   │
Authorization
   │
Application Layer
   │
Infrastructure
   │
PostgreSQL
```

Все соединения между компонентами защищены.

---

# 4. Аутентификация

Поддерживаются:

* регистрация по номеру телефона;
* OTP-код;
* JWT Access Token;
* Refresh Token;
* повторная авторизация при смене устройства.

---

# 5. JWT

Используются два типа токенов:

Access Token

* срок жизни 15 минут;
* используется для доступа к API.

Refresh Token

* срок жизни 30 дней;
* хранится отдельно;
* может быть отозван.

JWT содержит:

* user_id;
* account_id;
* role;
* permissions;
* token_version;
* exp;
* iat;
* jti.

---

# 6. Refresh Token

Refresh Token хранится:

* в PostgreSQL;
* в виде хеша;
* с информацией об устройстве;
* с IP-адресом;
* с User-Agent;
* с датой последнего использования.

При выходе пользователя токен отзывается.

---

# 7. OTP

Используется для:

* регистрации;
* входа;
* смены номера телефона;
* восстановления доступа.

Правила:

* срок действия — 5 минут;
* одноразовое использование;
* ограничение количества попыток;
* защита от повторной отправки.

---

# 8. Авторизация

Используется модель RBAC.

Основные роли:

* User;
* Moderator;
* Administrator;
* Super Administrator.

Каждая роль имеет набор разрешений.

---

# 9. Permissions

Разрешения назначаются независимо от ролей.

Примеры:

* profile.read;
* profile.update;
* meeting.create;
* event.delete;
* payment.refund;
* moderation.ban.

---

# 10. Политика паролей

Если в будущем будет добавлена авторизация по паролю:

* минимум 12 символов;
* Argon2id для хеширования;
* запрет повторного использования;
* защита от словарных паролей.

---

# 11. Защита API

Каждый запрос проходит:

1. Проверку HTTPS.
2. Проверку JWT.
3. Проверку роли.
4. Проверку разрешений.
5. Проверку Rate Limit.
6. Проверку состояния аккаунта.

---

# 12. Защита WebSocket

Перед открытием соединения выполняется:

* проверка JWT;
* проверка срока действия токена;
* проверка доступа к ресурсу.

Соединение автоматически закрывается при отзыве токена.

---

# 13. Защита PostgreSQL

Используются:

* отдельный пользователь БД для приложения;
* минимальные права;
* SSL-соединение;
* ограничения на сетевой доступ;
* резервное копирование;
* аудит операций.

Запрещается подключение к базе под суперпользователем.

---

# 14. Защита Redis

Redis используется только как:

* кэш;
* очередь;
* временное хранилище.

Запрещено хранить:

* пароли;
* номера карт;
* персональные документы.

---

# 15. Защита файлового хранилища

Файлы пользователей хранятся в MinIO/S3.

Требования:

* уникальные имена объектов;
* приватные бакеты;
* доступ по временным подписанным URL;
* проверка MIME-типа;
* ограничение размера файлов;
* антивирусная проверка перед публикацией.

---

# 16. Защита платежей

Приложение не хранит данные банковских карт.

Все операции выполняются через платёжный шлюз.

Хранятся только:

* идентификатор транзакции;
* статус;
* сумма;
* валюта;
* время проведения.

---

# 17. Логирование

Логируются:

* вход пользователя;
* выход пользователя;
* ошибки авторизации;
* создание и отзыв токенов;
* изменения ролей;
* административные действия;
* попытки несанкционированного доступа.

Пароли, OTP и содержимое сообщений в логах не сохраняются.

---

# 18. Audit Trail

Все критические действия записываются в журнал аудита.

Фиксируются:

* пользователь;
* время;
* IP-адрес;
* устройство;
* действие;
* результат.

---

# 19. Rate Limiting

Ограничиваются:

* вход;
* регистрация;
* отправка OTP;
* создание сообщений;
* создание мероприятий;
* загрузка файлов.

---

# 20. Защита от атак

Предусмотрена защита от:

* SQL Injection;
* XSS;
* CSRF (для web-клиента при использовании cookie);
* Clickjacking;
* Brute Force;
* Replay Attack;
* Token Theft;
* Path Traversal;
* SSRF (при работе с внешними URL);
* загрузки вредоносных файлов.

---

# 21. Работа с персональными данными

Персональные данные:

* минимизируются;
* передаются только по HTTPS;
* доступны только авторизованным пользователям;
* удаляются или анонимизируются по установленной политике хранения.

---

# 22. Управление секретами

Все секреты хранятся вне исходного кода.

Используются:

* переменные окружения;
* Docker Secrets (в production);
* внешние менеджеры секретов (при необходимости).

Запрещается хранить ключи в репозитории.

---

# 23. Мониторинг безопасности

Контролируются:

* неудачные попытки входа;
* аномальная активность;
* массовые запросы;
* изменения прав доступа;
* ошибки авторизации;
* подозрительные действия пользователей.

---

# 24. Проверки безопасности

Перед каждым релизом выполняются:

* статический анализ кода;
* проверка зависимостей;
* анализ известных уязвимостей;
* тестирование авторизации;
* тестирование прав доступа;
* проверка конфигурации Docker и PostgreSQL.

---

# 25. Политика разработки

Разработчикам запрещается:

* хранить секреты в Git;
* отключать проверку JWT;
* выполнять SQL-запросы без параметризации;
* передавать чувствительные данные в логах;
* обходить слой авторизации.

Все изменения, влияющие на безопасность, проходят обязательный код-ревью.

---

# 26. Соответствие архитектуре

Security Architecture интегрируется со всеми слоями системы:

* Domain Layer — бизнес-правила безопасности;
* Application Layer — проверка сценариев и разрешений;
* Infrastructure Layer — реализация механизмов защиты;
* API Layer — аутентификация и авторизация;
* PostgreSQL — контроль доступа и аудит.
