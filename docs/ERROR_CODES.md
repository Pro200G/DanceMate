# ERROR_CODES.md

# Коды ошибок API

**Проект:** DanceMate
**Версия:** 1.0

---

## 1. Назначение документа

Документ определяет единую систему кодов ошибок для API DanceMate.

Цели:
- единый формат ошибок для всех клиентов;
- понятная диагностика проблем;
- упрощение интеграции с фронтендом;
- централизованное управление ошибками;
- безопасное информирование пользователя.

Все ошибки API возвращаются в едином формате:

```json
{
  "success": false,
  "error": {
    "code": "ERROR_CODE",
    "message": "Понятное сообщение для пользователя",
    "details": null
  }
}
```

---

## 2. Принципы кодирования ошибок

### 2.1 Структура кода ошибки

```
DOMAIN_ACTION_REASON
```

- **DOMAIN** — контекст ошибки (3-4 заглавные буквы).
- **ACTION** — операция, в которой произошла ошибка.
- **REASON** — причина ошибки.

### 2.2 Примеры

```
AUTH_LOGIN_INVALID_CREDENTIALS
PROFILE_UPDATE_NOT_FOUND
EVENT_CREATE_FORBIDDEN
PAYMENT_PROCESS_DECLINED
```

### 2.3 HTTP статусы

Каждая ошибка маппится на соответствующий HTTP статус:

| Диапазон | Категория |
|----------|-----------|
| 4xx | Ошибка клиента |
| 5xx | Ошибка сервера |

## 16. Правила работы с ошибками

1. **Все ошибки API должны быть документированы** в этом файле перед добавлением в код.
2. **Новые коды ошибок** создаются по шаблону `DOMAIN_ACTION_REASON`.
3. **Сообщения для пользователя** должны быть понятными и не содержать технических деталей.
4. **Детали ошибки** могут содержать дополнительную техническую информацию для разработчиков (например, поле, в котором произошла ошибка).
5. **Логирование ошибок** выполняется на русском и английском (в формате JSON).
6. **Изменение существующих кодов ошибок** запрещено. Вместо этого создаётся новый код, а старый помечается как `DEPRECATED`.
7. **Удаление кодов ошибок** запрещено. Код может быть только помечен как `DEPRECATED`.

## 17. Добавление новой ошибки

Перед добавлением новой ошибки необходимо:

1. Проверить, не существует ли уже подходящий код в этом документе.
2. Если подходящего кода нет, создать новый по шаблону.
3. Обновить этот документ.
4. Добавить обработку ошибки в соответствующий Use Case.
5. Добавить тест для новой ошибки.
