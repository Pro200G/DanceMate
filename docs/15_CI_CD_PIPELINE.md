# CI/CD Pipeline

**Проект:** DanceMate

**Версия:** 1.0

---

# 1. Назначение

Документ определяет архитектуру непрерывной интеграции и доставки (CI/CD) проекта DanceMate.

Основные цели:

* автоматизация сборки;
* автоматизация тестирования;
* контроль качества кода;
* безопасное развертывание;
* сокращение времени выпуска новых версий;
* возможность быстрого отката изменений.

---

# 2. Общая схема CI/CD

```text
Developer
    │
    ▼
Git Commit
    │
    ▼
GitHub Repository
    │
    ▼
GitHub Actions (CI)
    │
    ├── Проверка форматирования
    ├── Статический анализ
    ├── Проверка типов
    ├── Unit Tests
    ├── Integration Tests
    ├── Security Scan
    ├── Проверка миграций
    ├── Сборка Docker
    │
    ▼
Docker Registry
    │
    ▼
Deployment Pipeline
    │
    ├── Development
    ├── Staging
    └── Production
```

---

# 3. Основные принципы

Pipeline должен обеспечивать:

* автоматическую проверку каждого изменения;
* воспроизводимость сборки;
* неизменяемость артефактов;
* автоматическое тестирование;
* автоматическое развёртывание;
* безопасный откат.

---

# 5. Workflow GitHub Actions

Каталог:

```text
.github/
└── workflows/
    ├── ci.yml
    ├── cd-development.yml
    ├── cd-staging.yml
    ├── cd-production.yml
    ├── docker-build.yml
    ├── release.yml
    ├── security.yml
    ├── dependency-update.yml
    └── documentation.yml
```

Каждый workflow отвечает за отдельный этап жизненного цикла.

---

# 6. Pipeline непрерывной интеграции (CI)

При каждом Push и Pull Request выполняются:

1. Проверка форматирования (`Black`, `isort`).
2. Анализ стиля (`Ruff`).
3. Проверка типов (`mypy`).
4. Проверка безопасности (`Bandit`).
5. Проверка зависимостей.
6. Unit Tests.
7. Integration Tests.
8. API Tests.
9. Проверка миграций Alembic.
10. Сборка Docker-образов.

При любой ошибке сборка завершается с ошибкой.

---

# 7. Проверка качества

Минимальные требования:

* покрытие кода ≥ 90%;
* отсутствуют ошибки линтеров;
* отсутствуют ошибки типизации;
* отсутствуют критические уязвимости;
* миграции применяются без ошибок.

---

# 8. Сборка Docker

После успешного прохождения CI выполняется:

* сборка образа Backend;
* присвоение тегов;
* публикация в реестр контейнеров.

Пример тегов:

```text
latest
develop
staging
v1.2.0
```

Образы должны быть неизменяемыми.

---

# 9. Развёртывание Development

Автоматически запускается после успешного объединения изменений в ветку `develop`.

Порядок:

1. Загрузка образов.
2. Запуск контейнеров.
3. Выполнение миграций.
4. Проверка состояния сервисов.
5. Smoke Tests.

---

# 10. Развёртывание Staging

Выполняется при создании ветки `release/*`.

Этапы:

1. Развёртывание новой версии.
2. Выполнение миграций.
3. Интеграционные тесты.
4. Нагрузочные проверки.
5. Проверка безопасности.

Staging максимально соответствует Production.

---

# 11. Развёртывание Production

Выполняется только после создания релиза.

Порядок:

1. Создание резервной копии PostgreSQL.
2. Создание резервной копии MinIO.
3. Проверка готовности окружения.
4. Остановка старой версии (при необходимости).
5. Запуск новой версии.
6. Выполнение миграций.
7. Health Checks.
8. Smoke Tests.
9. Публикация Release Notes.

---

# 12. Стратегии развёртывания

Поддерживаются:

* Rolling Deployment;
* Blue-Green Deployment.

Canary Deployment может быть добавлен в будущем.

---

# 13. Работа с миграциями

Все изменения схемы базы данных выполняются только через Alembic.

Pipeline:

1. Проверка наличия миграций.
2. Применение миграций.
3. Проверка схемы.
4. Выполнение тестов.

При ошибке миграции развёртывание отменяется.

---

# 14. Работа с секретами

Все секреты хранятся в GitHub Secrets.

Примеры:

* `DATABASE_URL`;
* `REDIS_URL`;
* `MINIO_ACCESS_KEY`;
* `MINIO_SECRET_KEY`;
* `JWT_SECRET_KEY`;
* `SMTP_PASSWORD`;
* `GHCR_TOKEN`.

Запрещается хранить секреты в репозитории.

---

# 15. Контроль безопасности

Перед развёртыванием выполняются:

* проверка зависимостей на известные уязвимости;
* анализ контейнеров;
* анализ Dockerfile;
* проверка конфигурации.

При обнаружении критических проблем выпуск блокируется.

---

# 16. Мониторинг после развёртывания

После публикации новой версии автоматически проверяются:

* доступность API;
* подключение к PostgreSQL;
* подключение к Redis;
* доступность MinIO;
* состояние WebSocket;
* время ответа.

---

# 17. Политика отката

Откат выполняется автоматически или вручную при:

* неуспешных Health Checks;
* критических ошибках миграций;
* недоступности сервисов.

Откат включает:

1. Возврат предыдущего Docker-образа.
2. Восстановление резервной копии базы данных (если требуется).
3. Повторную проверку работоспособности.

---

# 18. Уведомления

Pipeline отправляет уведомления о:

* успешной сборке;
* ошибках тестирования;
* успешном развёртывании;
* неудачном развёртывании;
* откате версии.

Каналы уведомлений:

* Email;
* Telegram;
* Slack (при необходимости).

---

# 19. Политика версий

Каждый успешный релиз:

* получает Git Tag;
* публикует Docker-образ;
* формирует Release Notes;
* обновляет номер версии приложения.

---

# 20. Требования к релизу

Перед выпуском должны быть выполнены:

* все проверки CI;
* успешное прохождение тестов;
* проверка миграций;
* проверка безопасности;
* успешное развёртывание в Staging;
* утверждение релиза ответственным разработчиком.

---

# 22. Метрики CI/CD

Контролируются:

* среднее время сборки;
* среднее время тестирования;
* частота успешных сборок;
* частота откатов;
* время восстановления после сбоя (MTTR);
* частота развёртываний.

Эти показатели регулярно анализируются для улучшения процесса.
